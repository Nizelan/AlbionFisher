# AlbionFisher — Специфікація

> Версія 0.1 · 2026-07-14 · Статус: чернетка для затвердження
> Джерело вимог: опис власника проєкту. Це опорний документ для architect / developer / verifier.

## 1. Огляд

**AlbionFisher** — зовнішня десктопна програма (Windows), яка автоматизує рибалку в грі **Albion Online**:

- захоплює зображення вибраного вікна гри;
- розпізнає ігрові об'єкти за допомогою **YOLO**-моделі (натренованої власником на своїх даних);
- керує мишею/клавіатурою, щоб виконувати цикл рибалки;
- має графічний інтерфейс на **Python**.

Програма **не** читає пам'ять гри і **не** втручається в її процес — лише скріншоти екрана та емуляція вводу (computer-vision підхід).

## 2. Цілі та не-цілі

### Цілі (MVP)

1. Вибір вікна гри Albion Online зі списку відкритих вікон.
2. Повний автоматичний цикл рибалки (розділ 5).
3. GUI: старт/стоп, вибір вікна, статус циклу, лічильники, лог подій.
4. Робота з YOLO-моделлю, яку користувач кладе у папку `model/`.

### Не-цілі (поза scope MVP)

- Переміщення персонажа по світу / пошук водойм на мапі.
- Продаж риби, робота з інвентарем, наживки/їжа.
- Injection, читання пам'яті, обхід анти-читу.
- Підтримка macOS/Linux.

## 3. Архітектура (модулі)

```
albionfisher/
  capture/     — захоплення кадрів вікна гри (швидкий screen capture)
  detection/   — YOLO inference: кадр → список об'єктів (class, bbox, conf)
  control/     — емуляція вводу: рухи миші, кліки, утримання кнопки
  fsm/         — state machine циклу рибалки (розділ 5)
  ui/          — GUI: налаштування, статус, лог
  config/      — налаштування користувача (json/yaml)
model/         — модель користувача: albionfisher.pt + classes.yaml
```

| Модуль | Рекомендовані бібліотеки (фінальний вибір — за architect) |
|--------|------------------------------------------------------------|
| capture | `mss` або `dxcam` (швидше для ігор), `pywin32` для пошуку вікна |
| detection | `ultralytics` (YOLOv8+), `opencv-python`, `torch` |
| control | `pydirectinput` / `pynput` (DirectInput-сумісний ввід для гри) |
| ui | `PySide6` (перевага) або `Tkinter` |
| runtime | Python 3.11+, Windows 10/11 |

## 4. Контракт YOLO-моделі

Користувач тренує модель самостійно і кладе файли в папку `model/`:

| Файл | Призначення |
|------|-------------|
| `model/albionfisher.pt` | Ваги натренованої YOLO-моделі (формат ultralytics) |
| `model/classes.yaml` | Перелік класів об'єктів — **єдине джерело правди** для тренування і програми |

Програма при старті звіряє класи моделі з `classes.yaml` і попереджає про розбіжності.

### Класи об'єктів для розпізнавання

| ID | Клас | Що це в грі | Де використовується в циклі |
|----|------|-------------|------------------------------|
| 0 | `fishing_zone` | Ділянка водойми з рибою (сплески/кола на воді) | Пошук місця закидання |
| 1 | `bobber_idle` | Поплавок на воді у стані очікування | Підтвердження закидання, очікування клювання |
| 2 | `bobber_bite` | Поплавок у момент клювання (занурення/сплеск) | Тригер підсікання |
| 3 | `minigame_bar` | Шкала міні-гри витягування | Виявлення старту міні-гри, межі шкали |
| 4 | `minigame_float` | Рухомий поплавок усередині шкали | Позиція для утримання по центру |
| 5 | `minigame_zone` | Цільова (центральна) зона шкали | Ціль утримання поплавка |
| 6 | `catch_popup` | Вікно/іконка результату улову | Підтвердження завершення циклу |

Повний файл: [`model/classes.yaml`](../model/classes.yaml). Зміни в класах — тільки через оновлення цього файлу + запис у `progress.md`.

## 5. Цикл рибалки (state machine)

```
IDLE → FIND_ZONE → CAST → WAIT_BITE → HOOK → MINIGAME → RESULT → FIND_ZONE (loop)
                                  ↘ timeout ↘ fail → RECAST/IDLE
```

| Стан | Дія програми | Перехід далі | Перехід при збої |
|------|--------------|--------------|------------------|
| `IDLE` | Очікує кнопку Start; перевіряє вікно і модель | Start → `FIND_ZONE` | — |
| `FIND_ZONE` | Шукає `fishing_zone` у кадрі | Знайдено → `CAST` | Не знайдено N сек → пауза + повторити, звіт у UI |
| `CAST` | Наводить курсор на зону, затискає ЛКМ на силу закиду, відпускає | `bobber_idle` виявлено → `WAIT_BITE` | Поплавок не з'явився за T сек → повторити CAST (макс. 3) |
| `WAIT_BITE` | Стежить за поплавком | `bobber_bite` → `HOOK` | Таймаут клювання (config, ~30 с) → перезакид |
| `HOOK` | Миттєвий клік ЛКМ (підсікання) | `minigame_bar` виявлено → `MINIGAME` | Міні-гра не почалась за T сек → `RESULT` (пуста) |
| `MINIGAME` | Утримує/відпускає ЛКМ, щоб `minigame_float` лишався в `minigame_zone` (керування: hold → рух в один бік, release → в інший) | Шкала зникла → `RESULT` | Поплавок вийшов за межі → риба зірвалась → `RESULT` (fail) |
| `RESULT` | Фіксує улов/провал (через `catch_popup` або відсутність), оновлює лічильники | → `FIND_ZONE` | Стоп-умови (розділ 6) → `IDLE` |

### Керування в міні-грі

**Механіка (підтверджено власником, 2026-07-14):** поплавок (`minigame_float`) **завжди дрейфує ліворуч** сам по собі; **утримання ЛКМ рухає його праворуч**. Release = дрейф ліворуч.

- Частота реакції: обробка кадру + рішення ≥ 15 FPS (ціль 20–30).
- Алгоритм MVP: bang-bang контролер з гістерезисом —
  - центр `minigame_float` **лівіше** центру `minigame_zone` → **hold ЛКМ** (рух праворуч);
  - центр `minigame_float` **правіше** центру `minigame_zone` → **release** (дрейф ліворуч);
  - у межах deadzone навколо центру — не змінювати стан кнопки (проти «тремтіння»).
- Параметри (deadzone, затримки) — у конфізі, підбираються на практиці.

## 6. Вимоги до GUI (Python)

1. **Вибір вікна гри** — випадний список відкритих вікон (фільтр «Albion Online»), кнопка Refresh.
2. **Start / Stop** — запуск і зупинка циклу; гаряча клавіша аварійної зупинки (напр. `F10`) працює глобально.
3. **Статус** — поточний стан FSM, час у стані, FPS розпізнавання.
4. **Лічильники** — впіймано / зірвалось / перезакиди, час сесії.
5. **Лог подій** — прокручуваний список з часовими мітками; дублюється у файл `logs/session_*.log`.
6. **Налаштування** — шлях до моделі, поріг confidence (default 0.5), таймаути, сила закиду, стоп-умови.
7. **(Nice-to-have)** прев'ю кадру з намальованими bbox — для налагодження моделі.

### Стоп-умови (fail-safes)

- Вікно гри закрите або згорнуте → пауза + повідомлення.
- N провалів поспіль (config, default 5) → автозупинка.
- Ліміт часу сесії / кількості риби (опційно, config).
- Аварійна клавіша → миттєва зупинка і відпускання всіх кнопок.

## 7. Нефункціональні вимоги

| Вимога | Значення |
|--------|----------|
| Затримка кадр→дія в міні-грі | ≤ 70 мс на середньому ПК з GPU; CPU-fallback допустимий з попередженням |
| Продуктивність поза міні-грою | Розпізнавання 2–5 FPS достатньо (економія ресурсів) |
| Стабільність | Безперервна сесія ≥ 1 год без крешів і витоків пам'яті |
| Конфігурація | Все, що «магічні числа» — у `config/settings.yaml` |
| Логи | Достатні для розбору «чому не впіймав» без запуску відладчика |

## 8. Критерії приймання (для verifier)

1. Програма запускається, знаходить вікно Albion Online, показує GUI.
2. Із валідною моделлю в `model/` статус детекції «OK»; з відсутньою/битою — зрозуміла помилка, без крешу.
3. На тестовому записі/стенді FSM проходить повний цикл: зона → закид → клювання → міні-гра → результат.
4. У міні-грі поплавок утримується в зоні ≥ 80% часу на еталонному записі.
5. Аварійна клавіша зупиняє програму з будь-якого стану ≤ 200 мс.
6. Лічильники та лог відповідають фактичним подіям тестового прогону.
7. Юніт-тести FSM (переходи, таймаути) проходять; лінт чистий.

## 9. Ризики та застереження

- **ToS Albion Online**: автоматизація геймплею порушує умови гри і може призвести до бану акаунта. Програма розробляється на відповідальність власника; у README — явне попередження.
- Патчі гри змінюють графіку → модель потребує перетренування (тому класи зафіксовані в `classes.yaml`).
- Анти-чит може детектити синтетичний ввід → ризик приймає власник.
- Якість моделі — поза контролем програми; програма лише репортить низький confidence.
